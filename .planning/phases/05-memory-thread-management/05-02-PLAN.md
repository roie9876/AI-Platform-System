<task type="auto">
  <name>Task 1: MemoryService — embed, store, retrieve</name>
  <files>
    backend/app/services/memory_service.py
  </files>
  <action>
  Create `backend/app/services/memory_service.py` with class `MemoryService`:

  1. **`async def embed(self, text: str, model_endpoint: ModelEndpoint) -> List[float]`**
     - Use `litellm.aembedding()` to generate embeddings.
     - Build the LiteLLM model string from the endpoint's provider_type/model_name (same pattern as ModelAbstractionService).
     - Use `text-embedding-3-small` as the embedding model name (pass via parameter or default).
     - Handle errors gracefully — return empty list on failure (memories still saved without embedding).

  2. **`async def store_memory(self, agent_id, user_id, tenant_id, content, db, model_endpoint=None, memory_type="knowledge", source_thread_id=None)`**
     - Create AgentMemory record.
     - If model_endpoint provided, generate embedding via `embed()` and set on the record.
     - Commit and return the memory record.

  3. **`async def retrieve_relevant(self, query: str, agent_id, user_id, tenant_id, db, model_endpoint=None, top_k=5) -> List[AgentMemory]`**
     - If model_endpoint provided, generate embedding for query.
     - Query AgentMemory filtered by agent_id + user_id + tenant_id.
     - If embedding available, order by cosine distance: `AgentMemory.embedding.cosine_distance(query_embedding)`.
     - If no embedding, fall back to returning most recent memories.
     - Return top_k results.

  4. **`async def extract_memories_from_thread(self, thread_id, agent_id, user_id, tenant_id, db, model_endpoint=None)`**
     - Load all assistant messages from the thread.
     - For each substantive assistant message (>50 chars), store as a long-term memory with source_thread_id set.
     - This is called optionally (e.g., when a thread is closed or on-demand).

  Use the existing pattern from ModelAbstractionService for building LiteLLM params. The embedding endpoint can be derived from the agent's assigned model_endpoint — use the same provider credentials but with an embedding model name.
  </action>

  </verify>
  <done>MemoryService can generate embeddings via LiteLLM, store memories with vectors in pgvector, and retrieve relevant memories by cosine similarity.</done>

  <name>Task 2: Integrate thread persistence + memory into agent execution</name>
  <files>

  <action>
  1. **Update `ChatRequest` schema** in schemas.py:
     - Add `thread_id: Optional[UUID] = None` field. When provided, messages are persisted to this thread and history is loaded from the thread (ignoring `conversation_history`).

  2. **Update `chat.py` endpoint**:
     - If `body.thread_id` is provided:
       a. Load the Thread (validate ownership: tenant_id + user_id).
       b. Load ThreadMessages ordered by sequence_number as the conversation_history (replacing client-sent history).
       c. Pass thread_id to the execution service.
     - If no thread_id, behavior is unchanged (stateless, client-managed history).

  3. **Update `agent_execution.py` `execute()` method**:
     - Add `thread_id: Optional[UUID] = None` and `user_id: Optional[UUID] = None` parameters.
     - **Before streaming**: If thread_id provided, save the user message as a ThreadMessage (compute next sequence_number from max existing + 1).
     - **After streaming completes**: If thread_id provided, save the assistant response as a ThreadMessage with message_metadata containing sources if any.
     - **Auto-title**: If thread is new (only 1 user message), set thread.title to first 80 chars of the user message.
     - **Execution logging**: For each significant event (message_sent, model_response, tool_call), create an ExecutionLog entry with event_type, state_snapshot (messages count, tool names), and timing.

  4. **Memory injection in prompt construction**:
     - After loading RAG context and before model call, if thread_id is provided:
       a. Call `MemoryService.retrieve_relevant()` with the user message.
       b. If relevant memories found, inject as a system message: "Relevant memories from past interactions: ..."
       c. Insert after RAG context but before conversation history.

  Keep backward compatibility: when thread_id is None, everything works exactly as before (stateless mode).
  </action>
