**TenantProvisioningService** (backend/app/services/tenant_provisioning.py):

Create a service class that handles the full provisioning pipeline for a new tenant:

- `provision_tenant(tenant: dict)` — Orchestrates the full provisioning sequence:
  1. Call `_provision_k8s_namespace(tenant["slug"])` 
  2. Call `_seed_default_data(tenant["id"])`
  3. Call `_create_admin_user(tenant["id"], tenant["admin_email"])`
  4. Transition tenant state from "provisioning" → "active" via TenantService

- `_provision_k8s_namespace(slug: str)` — Execute `k8s/scripts/setup-tenant.sh {slug}` via `asyncio.create_subprocess_exec`. Log stdout/stderr. Raise RuntimeError if exit code != 0. Use `shlex.quote(slug)` to sanitize the slug input (prevent shell injection). Find the script path relative to the project root using `pathlib.Path(__file__).parent.parent.parent.parent / "k8s" / "scripts" / "setup-tenant.sh"`.

- `_seed_default_data(tenant_id: str)` — Create seed data using existing repositories:
  - Import `ToolRepository` and create 2 default built-in tools:
    - `{"name": "web_search", "type": "built-in", "description": "Search the web", "config": {}}`
    - `{"name": "code_interpreter", "type": "built-in", "description": "Execute Python code", "config": {}}`
  - Import `CatalogRepository` and create 1 default catalog entry:
    - `{"name": "Getting Started", "type": "template", "description": "Default agent template", "config": {"system_prompt": "You are a helpful assistant."}}`
  - Each item gets `tenant_id` set by the repository's `create()` method.

- `_create_admin_user(tenant_id: str, admin_email: str)` — Use `UserRepository` to create an admin user:
  - `{"email": admin_email, "full_name": "Tenant Admin", "role": "admin", "is_active": True}`
  - Check if user already exists via `get_by_email` first — skip if exists.

**Update TenantService.create_tenant** (backend/app/services/tenant_service.py):

- Add `import logging` and `logger = logging.getLogger(__name__)` if not present.

**Enhance TenantMiddleware** (backend/app/middleware/tenant.py):

After extracting tenant_id from the Entra token (existing logic), add a tenant status check:

1. Import `TenantRepository` at module level.
2. After `request.state.tenant_id` is set, query the tenant record:
   ```python
   tenant_repo = TenantRepository()
   tenant = await tenant_repo.get(tenant_id, tenant_id)  # self-partitioned
   ```
3. If tenant is `None` → return 403 JSON response: `{"detail": "Tenant not found"}`
4. If tenant `status` is `"suspended"` → return 403 JSON response: `{"detail": "Tenant is suspended. Contact platform administrator."}`
5. If tenant `status` is `"deactivated"` → return 403 JSON response: `{"detail": "Tenant is deactivated. Contact platform administrator."}`
6. If tenant `status` is `"deleted"` → return 403 JSON response: `{"detail": "Tenant no longer exists."}`
7. If tenant `status` is `"provisioning"` → return 503 JSON response: `{"detail": "Tenant is being provisioned. Please try again shortly."}`
8. Only `"active"` status proceeds normally.

**Performance note:** Add a simple in-memory TTL cache for tenant status to avoid hitting Cosmos DB on every request. Use a module-level dict `_tenant_status_cache: dict[str, tuple[str, float]] = {}` with 60-second TTL. Check cache first, only query Cosmos DB on cache miss or expiry. This is acceptable for suspension blocking — a 60-second delay between suspension and enforcement is fine.

Use `from starlette.responses import JSONResponse` to return error responses directly instead of calling `call_next`.

**Important:** The /api/v1/tenants/* endpoints (tenant management) should bypass this check since platform admins manage tenants cross-tenant. Add `/api/v1/tenants` to the PUBLIC_PATHS or add a separate check: if the path starts with `/api/v1/tenants` → skip tenant status check (platform admin endpoints have their own role check).
