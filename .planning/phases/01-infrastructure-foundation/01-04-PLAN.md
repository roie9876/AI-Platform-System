    **1. .github/workflows/infra-deploy.yml** — Infrastructure deployment pipeline per D-16/D-17:

    ```yaml
    name: Deploy Infrastructure
    ```

    **Triggers:**
    - `push` to `main` branch, paths: `infra/**`
    - `pull_request` to `main` branch, paths: `infra/**` (for what-if preview)
    - `workflow_dispatch` with environment input (dev/staging/prod)

    **Permissions** (OIDC federation per D-17):
    ```yaml
    permissions:
      id-token: write
      contents: read
    ```

    **Environment variables:**
    - `AZURE_SUBSCRIPTION_ID` — from GitHub environment secrets
    - `AZURE_TENANT_ID` — from GitHub environment secrets
    - `AZURE_CLIENT_ID` — from GitHub environment secrets (OIDC app registration)

    **Jobs:**

    **Job 1: validate**
    - Runs on: `ubuntu-latest`
    - Steps:
      1. Checkout code
      2. Azure Login with OIDC (`azure/login@v2` with `client-id`, `tenant-id`, `subscription-id`)
      3. Install Bicep CLI: `az bicep install`
      4. Run validation script: `bash infra/scripts/validate.sh`
      5. Lint: `az bicep lint --file infra/main.bicep`

    **Job 2: preview** (on pull_request only)
    - Needs: validate
    - Condition: `github.event_name == 'pull_request'`
    - Steps:
      1. Checkout, Azure Login (same as validate)
      2. What-if: `az deployment sub what-if --location eastus2 --template-file infra/main.bicep --parameters infra/parameters/dev.bicepparam tenantId=$AZURE_TENANT_ID`
      3. Post what-if output as PR comment using `actions/github-script@v7`

    **Job 3: deploy-dev**
    - Needs: validate
    - Condition: `github.ref == 'refs/heads/main' && github.event_name == 'push'`
    - Environment: `dev` (GitHub environment with protection rules)
    - Steps:
      1. Checkout, Azure Login
      2. Deploy: `az deployment sub create --location eastus2 --template-file infra/main.bicep --parameters infra/parameters/dev.bicepparam tenantId=$AZURE_TENANT_ID --name "infra-$(date +%Y%m%d-%H%M%S)"`

    **Job 4: deploy-staging** (manual, depends on dev success)
    - Needs: deploy-dev
    - Condition: `github.event_name == 'workflow_dispatch' && github.event.inputs.environment == 'staging'`
    - Environment: `staging` (with required reviewers)
    - Same deploy pattern with `staging.bicepparam`

    **Job 5: deploy-prod** (manual, depends on staging)
    - Needs: deploy-staging
    - Condition: `github.event_name == 'workflow_dispatch' && github.event.inputs.environment == 'prod'`
    - Environment: `production` (with required reviewers + wait timer)
    - Same deploy pattern with `prod.bicepparam`

    **2. infra/scripts/validate.sh** — Bicep validation (used by CI and local dev):
    ```bash
    #!/usr/bin/env bash
    set -euo pipefail

    echo "=== Bicep Build (compile check) ==="
    az bicep build --file infra/main.bicep --stdout > /dev/null

    echo "=== Bicep Lint ==="
    az bicep lint --file infra/main.bicep

    echo "=== Parameter file validation ==="
    for paramfile in infra/parameters/*.bicepparam; do
      echo "Validating: $paramfile"
      az bicep build-params --file "$paramfile" --stdout > /dev/null
    done

    echo "=== All validations passed ==="
    ```
    Make the script executable: `chmod +x infra/scripts/validate.sh`

  <action>
    Create `.github/workflows/service-deploy.yml` — Service build, push, and deploy pipeline:

    ```yaml
    name: Build and Deploy Services
    ```

    **Triggers:**
    - `push` to `main`, paths: `apps/**`, `packages/**`
    - `pull_request` to `main`, paths: `apps/**`, `packages/**` (build only, no deploy)
    - `workflow_dispatch` with inputs:
      - `service` (string, required — service directory name under apps/)
      - `environment` (choice: dev/staging/prod)

    **Permissions:**
    ```yaml
    permissions:
      id-token: write
      contents: read
      packages: write
    ```

    **Environment variables:**
    - `AZURE_SUBSCRIPTION_ID`, `AZURE_TENANT_ID`, `AZURE_CLIENT_ID` — from secrets
    - `ACR_LOGIN_SERVER` — from GitHub environment variables (set after infra deployed)
    - `AKS_CLUSTER_NAME` — from GitHub environment variables
    - `AKS_RESOURCE_GROUP` — from GitHub environment variables

    **Jobs:**

    **Job 1: detect-changes**
    - Uses `dorny/paths-filter@v3` to detect which apps/ directories changed
    - Outputs a matrix of changed services
    - On `workflow_dispatch`, uses the input service name directly

    **Job 2: build** (matrix strategy based on detect-changes output)
    - Runs on: `ubuntu-latest`
    - Steps:
      1. Checkout
      2. Setup Node.js 22 (`actions/setup-node@v4`)
      3. Setup pnpm (`pnpm/action-setup@v4`)
      4. Install dependencies: `pnpm install --frozen-lockfile`
      5. Lint: `pnpm turbo lint --filter=${{ matrix.service }}`
      6. Test: `pnpm turbo test --filter=${{ matrix.service }}`
      7. Build: `pnpm turbo build --filter=${{ matrix.service }}`

    **Job 3: push-image** (on main branch push or workflow_dispatch only)
    - Needs: build
    - Condition: `github.ref == 'refs/heads/main'`
    - Steps:
      1. Checkout
      2. Azure Login (OIDC)
      3. ACR Login: `az acr login --name $ACR_LOGIN_SERVER`
      4. Docker build and push using `docker/build-push-action@v5`:
         - Context: `apps/${{ matrix.service }}`
         - Tags: `$ACR_LOGIN_SERVER/${{ matrix.service }}:${{ github.sha }}`, `$ACR_LOGIN_SERVER/${{ matrix.service }}:latest`
         - Push: true

    **Job 4: deploy-dev**
    - Needs: push-image
    - Environment: `dev`
    - Steps:
      1. Azure Login (OIDC)
      2. Get AKS credentials: `az aks get-credentials --resource-group $AKS_RESOURCE_GROUP --name $AKS_CLUSTER_NAME`
      3. Deploy via kubectl or Helm:
         ```
         kubectl set image deployment/${{ matrix.service }} \
           ${{ matrix.service }}=$ACR_LOGIN_SERVER/${{ matrix.service }}:${{ github.sha }} \
           --namespace ai-platform
         ```
      4. Wait for rollout: `kubectl rollout status deployment/${{ matrix.service }} --namespace ai-platform --timeout=300s`

    **Note:** Helm charts will be created in a later phase. For now, use kubectl set image for simplicity. The workflow is structured to be easily upgraded to Helm later.
