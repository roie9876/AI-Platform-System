    **Required ADRs (minimum — add more if natural):**

    ADR-001: Python as Backend Language
    - Context: Need a language for API + agent runtime. AI/ML ecosystem, agent frameworks (LangChain, AutoGen, Semantic Kernel, CrewAI) are Python-first.
    - Decision: Python 3.12+ with FastAPI
    - Consequences: +AI ecosystem native, +async performance, -frontend/backend language mismatch

    ADR-002: FastAPI as API Framework
    - Context: Need async web framework with auto-generated API docs. Alternatives: Django REST, Flask, Litestar.
    - Decision: FastAPI with Pydantic validation
    - Consequences: +auto OpenAPI, +async native, +Pydantic integration, -smaller community than Django

    ADR-003: PostgreSQL + pgvector Over Separate Vector DB
    - Context: Need relational storage AND vector storage for RAG. Options: PostgreSQL+pgvector vs PostgreSQL+Pinecone vs PostgreSQL+Weaviate.
    - Decision: PostgreSQL 16+ with pgvector extension
    - Consequences: +single database to manage, +reduced infra complexity, +sufficient for PoC scale, -may need dedicated vector DB at extreme scale

    ADR-004: Semantic Kernel as Agent Framework
    - Context: Need agent orchestration framework. Options: Semantic Kernel, AutoGen, LangChain, CrewAI.
    - Decision: Semantic Kernel (Python SDK)
    - Consequences: +Microsoft-native alignment, +stable API, +plugin architecture maps to tool marketplace, -less community content than LangChain

    ADR-005: Multi-Tenancy via Row-Level Isolation
    - Context: Need tenant isolation strategy. Options: database-per-tenant, schema-per-tenant, row-level with tenant_id.
    - Decision: Row-level isolation with tenant_id in JWT, middleware-enforced filtering
    - Consequences: +simple infrastructure, +easy to add tenants, +cost efficient, -requires discipline in every query, -less isolation than DB-per-tenant

    ADR-006: Next.js with App Router for Frontend
    - Context: Need modern frontend framework for complex admin UI. Options: Next.js, Remix, Vite+React SPA.
    - Decision: Next.js 15+ with App Router, React 19, Shadcn/ui + Tailwind CSS
    - Consequences: +SSR/SSG for performance, +API routes for BFF, +large ecosystem, -complexity of App Router

    ADR-007: JWT in httpOnly Cookies Over Bearer Tokens
    - Context: Need auth token transport. Options: Bearer header, httpOnly cookie, session-based.
    - Decision: JWT in httpOnly cookies with access (30min) + refresh (7d) rotation
    - Consequences: +XSS protection, +automatic on every request, -requires CORS config, -slightly more complex than Bearer

    ADR-008: Azure-First Cloud Strategy
    - Context: Platform needs cloud deployment target. Company is Microsoft-aligned (STU-MSFT).
    - Decision: Azure as primary cloud with Microsoft services wherever possible
    - Consequences: +organizational alignment, +Entra ID integration, +Azure OpenAI default, -vendor lock-in to Microsoft ecosystem

    ADR-009: Model-Agnostic via OpenAI-Compatible Interface
    - Context: Platform must support multiple model providers. Options: vendor-specific SDKs, unified abstraction, OpenAI-compatible format.
    - Decision: OpenAI-compatible API format as universal interface, LiteLLM for provider abstraction
    - Consequences: +most providers support this format, +single abstraction, -some provider features may not map cleanly

    ADR-010: Celery + Redis for Async Task Execution
    - Context: Agent execution can be long-running. Need async task processing. Options: Celery, Dramatiq, Taskiq, in-process asyncio.
    - Decision: Celery with Redis as broker and result backend
    - Consequences: +proven at scale, +priority queues, +task chains, -operational complexity, -Redis as single point of failure

    **Part B: Add Inline Technology Comparisons** throughout the existing document sections per D-06.

    For each major architecture section where a technology choice was made, add a brief comparison callout. Use this format:

    > **Why {chosen}?** We chose {X} over {Y} because {Z}. {Y} was considered but {tradeoff}.

    Insert these inline comparisons in the relevant sections:

    1. In "System Overview" or "Architecture Layers — Control Plane" section:
       > **Why FastAPI?** We chose FastAPI over Django REST Framework because FastAPI provides native async support, automatic OpenAPI spec generation, and Pydantic-based validation — critical for an API-first platform. Django was considered but its synchronous-first model adds complexity for real-time agent streaming.

    2. In "Architecture Layers — Data Layer" section:
       > **Why pgvector over Pinecone/Weaviate?** We chose pgvector (PostgreSQL extension) over dedicated vector databases because it eliminates a separate infrastructure component while providing sufficient vector search performance for PoC scale. Pinecone offers better scaling but adds operational complexity and cost.

    3. In "Architecture Layers — Runtime Plane" near agent framework discussion:
       > **Why Semantic Kernel over LangChain?** We chose Semantic Kernel because it's Microsoft-native with a stable API and plugin architecture that maps directly to our tool marketplace concept. LangChain has a larger community but its frequent breaking changes and heavy abstraction layers make it harder to debug at scale.

    4. In "Security Architecture" section:
       > **Why httpOnly cookies over Bearer tokens?** We chose httpOnly cookies for JWT transport because they provide automatic XSS protection — the token is never accessible to JavaScript. Bearer tokens were considered but require manual token management in frontend code and are vulnerable to XSS attacks.

    5. In "Deployment Architecture" section:
       > **Why AKS over Azure Container Apps?** We chose AKS for production because multi-tenant workload isolation requires fine-grained control over pod scheduling, network policies, and namespace isolation. Azure Container Apps was considered for simplicity but lacks the namespace-level isolation needed for enterprise multi-tenancy.

    6. In "Microsoft Azure Service Mapping" section:
       > **Why Azure AI Search over Elasticsearch?** We chose Azure AI Search for RAG retrieval because it provides native hybrid (vector + keyword) search, integrates with Azure Blob Storage for document ingestion, and is fully managed. Elasticsearch was considered but requires more operational overhead and doesn't integrate as tightly with the Azure ecosystem.
